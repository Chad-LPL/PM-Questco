## Product intent

Submission process for completed workbook ingestion; likely done via triggering in workbook or via SharePoint triggering. This supports Taylor V1 open enrollment orchestration. Orchestration, reliability, and observability for renewal processing.

### April 8, 2026

**Workbook generation and regeneration** should not assume **ClientSpace** as the only trigger—evaluate **manual operator procedures**, **admin/Taylor UI**, and **Microsoft 365** automation. Same theme applies to **batch vs one-off** regeneration when rebanding or OMQ templates change ([2.3.6](./2.3.6-Admin-console-providing-renewal-status-visibility-and-light-reporting.md), [2.6.8.2](./2.6.8.2-Parallel-generation-of-one-some-or-all-client-workboos.md)).

### April 9, 2026 — executed workbook ingestion

- **Today:** Reps upload the executed workbook to **ClientSpace** and advance the batch; **no SharePoint** in that step.
- **Target:** Taylor must react when the **executed** file is ready—candidate triggers: **SharePoint** subscription/webhook, **global ingest folder** + **client ID** in workbook, or **ClientSpace → HTTP** callback to Taylor ([2.6.6](./2.6.6-Workbook-completion-trigger-automation.md)). **Taylor** posts the **stored file URL** back to ClientSpace so reps do not duplicate link maintenance ([2.6.5](./2.6.5-SharePoint-storage-and-posting-URL-to-ClientSpace.md)).
- **Re-uploads:** Design for **delete/replace** and **client changed mind** scenarios (validation blocks obviously blank/incomplete files per [2.6.7](./2.6.7-Retrieval-parsing-normalization-and-validation-of-executed-workbooks.md)).
