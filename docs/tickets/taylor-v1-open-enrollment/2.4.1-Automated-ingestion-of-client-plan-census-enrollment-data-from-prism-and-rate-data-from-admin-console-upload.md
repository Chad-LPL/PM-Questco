# 2.4.1 - Automated ingestion of client, plan, census, enrollment data from prism, and rate data from admin console upload (uploaded on demand)

## Traceability
- **Asana task ID:** `1213866553394534`
- **Section:** Backlog

## Product intent

Deliver automated ingestion of **client, plan, census, and enrollment** data from **Prism**, plus **rate** inputs needed for workbook generation—sourced from the **authoritative channel** Questco selects per [2.5.3](./2.5.3-Rate-book-management.md).

### Decision note — rate data source (April 2026)

The Asana title names **“rate data from admin console upload.”** Discovery with the product owner ([AnnaSync-WorkflowReview-4.7.26.md](../../meetings/AnnaSync-WorkflowReview-4.7.26.md)) favors **Prism-first** rates: **future rate band** and related fields may be populated via **Prism import/API** with **minimal redundant ClientSpace rate entry**. **Admin console upload** remains a **valid fallback** or parallel path until Questco (Anna/Dimple + **LaWanda**) confirms the lift of importing the carrier rate grid into Prism’s **future rate band**.

**Until that decision lands:** treat acceptance criteria as **(a)** Prism-sourced client/plan/census/enrollment **plus** **(b)** rate inputs from either **Prism** or **operator upload** (admin/Taylor), with the final split documented in [2.5.3](./2.5.3-Rate-book-management.md).

### Decision note — April 8, 2026

- **Future rates in Prism:** Questco aligned on importing **future rates** via the path **Kelly/Kim** (rebanding) → **LaWanda** (Prism import using the file/columns LaWanda confirmed with Dimple). Taylor should **read** finalized rates from Prism for workbook generation where available.
- **Band assignment:** **Client band ID** is **not** stored as an automation-friendly field in Prism; Taylor must **ingest the rebanding file** (see [2.5.3](./2.5.3-Rate-book-management.md)) from **SharePoint** ([2.2.3](./2.2.3-SharePoint-file-storage-configuration.md)) to know which band applies per client/carrier. The file lists **all clients**, not only changed bands; late band changes imply **rerunning** that client’s workbook generation.
- **Idempotent runs:** If upstream data changes, operators should be able to **regenerate** a single client or batch without manual cleanup—aligns with [2.3.3](./2.3.3-Error-handling-retries-and-exception-management.md) and [2.6.8.2](./2.6.8.2-Parallel-generation-of-one-some-or-all-client-workboos.md).

- Automated ingestion of client, plan, census, and enrollment data from **Prism** APIs (normalized per [2.2.5](./2.2.5-Adapter-mapping-layer-to-manage-Prism-schema-dependencies.md)). **Coverage of workbook surfaces** (`CSExport`, `ClassesPlans`, `CurrentPlans`, `RenewalPlans`, class tabs, optional `Census`) is defined with Questco in [Renewal-Workbook-Questco-Discovery.md](../../prd/Renewal-Workbook-Questco-Discovery.md) and implemented via the Prism → workbook mapping matrix ([2.2.5](./2.2.5-Adapter-mapping-layer-to-manage-Prism-schema-dependencies.md), [2.6.2](./2.6.2-Populate-workbooks-with-Prism-and-rate-book-data.md)).
- **Rate** ingestion path aligned to recorded decision: Prism-sourced and/or on-demand upload, with no unstated assumption that admin upload is the only route. Final authority per [2.5.3](./2.5.3-Rate-book-management.md).
- Completion can be verified without unstated assumptions: evidence aligns with the success criteria below.
- Errors, edge cases, and operator-visible states relevant to this scope are handled or explicitly deferred with tracking.
