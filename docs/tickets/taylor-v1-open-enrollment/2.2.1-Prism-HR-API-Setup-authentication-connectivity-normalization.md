# 2.2.1 - Prism HR API Setup (authentication, connectivity, normalization)

## Traceability
- **Asana task ID:** `1213834655988109`
- **Section:** Backlog

## Product intent

Authenticated access to Prism HR APIs for renewal data (client, plan, census, enrollment), including **re-read** after workbook work (Prism overwrites blanks on import) and support for a **full client list** ([2.2.6](./2.2.6-Client-list-synchronization-with-ClientSpace.md)). Responses are **normalized** so downstream code uses a stable internal model ([2.2.5](./2.2.5-Adapter-mapping-layer-to-manage-Prism-schema-dependencies.md)), not raw payloads.

**Credential model (Questco):** Distinguish (1) **interactive Prism 1ID** logins—**one per developer**, tied to email, with SMS/app MFA for human UI access—from (2) **web service / API user** credentials used by Taylor (app-to-app: **username/password**, typically **one per environment**, e.g. sandbox vs production, named e.g. for the Taylor integration). The API user is not subject to the 1ID SMS flow. Store secrets in Key Vault per [2.2.4](./2.2.4-Azure-hosting-architecture-and-environment-setup.md); document rotation.

## Product requirements (verifiable)

- **Session:** All API calls require **Create PEO Session** (or equivalent); pass returned **token in headers** for every subsequent call (Questco pattern).
- **Sandbox scope:** Non-prod access limited to agreed test **company/client IDs: 102, 104, 105, 107** (configure API user scoping when Prism allows). Treat as shared sandbox—coordinate destructive tests.
- Authenticate with agreed credentials; secrets in Key Vault; documented rotation.
- Classify failures in logs/errors: auth, network, HTTP/API, **schema drift** (unexpected shape).
- **Correlation IDs** on Prism calls.
- Use normalized model / adapter output on paths feeding [2.4.1](./2.4.1-Automated-ingestion-of-client-plan-census-enrollment-data-from-prism-and-rate-data-from-admin-console-upload.md).
- Redact or synthetic **PII** in logs and committed fixtures (align with MSA/DPA, [docs/README.md](../../README.md)).
- Retries/backoff for transient errors; document rate limits and tenant URLs when Prism confirms.
- Optional non-prod tests that catch breaking API changes.

### Integration notes (spike / implementation)

Priority endpoints called out by Questco: **GetClientMaster** (broad client data); benefits area—**Get Active Benefit Plans**, **Get Benefit Plans**, **Get Client Benefit Plans**, benefit rules as needed; **GetData** (e.g. system category) for **bulk ClientMaster** instead of per-client iteration. **Get Available Benefit Plans** overlaps naming with other benefit endpoints—**confirm usage** during spike (may be employee-scoped; not always required). Official Prism API documentation is limited; validate against real sandbox behavior.

## Success criteria

- [ ] Authenticated Prism call in non-production with **correlation id** in logs (using **PEO session** pattern).
- [ ] **Redacted fixtures** for regression tests from normalized responses.
- [ ] Documented **failure taxonomy** reflected in at least one structured error path.
- [ ] Runbook entry for rate limits, sandbox URLs, **web service user**, and **allowed test company IDs**.
