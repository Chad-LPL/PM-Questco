## Product intent

Deliver the capability described in the ticket title: Renewal state machine with defined milestones and transitions. This is part of Taylor V1 open enrollment orchestration. Orchestration, reliability, and observability for renewal processing.

### Product flow (confirmed 2026-04-07)

End-to-end spine as confirmed with the product owner ([AnnaSync-WorkflowReview-4.7.26.md](../../meetings/AnnaSync-WorkflowReview-4.7.26.md)):

- **Start:** Taylor begins from the **existing ClientSpace benefit batch**. There should be **one active batch per client** (prior year expired, current year labeled active). The user **renews** that batch—Taylor does not assume net-new batch creation as the only entry point.
- **System of record:** **Prism** is trusted for demographics, design, and benefit truth; payroll runs through Prism. ClientSpace can drift mid-year and is not authoritative for that data.
- **After batch renewal:** Taylor updates the new batch with current **Prism** data (renewal flags, effective date, plans offered, etc.). Then operations open the **BSR** (Benefit Setup and Renewal data form)—**new**, **auto-linked** to that batch—for broker type, renewal metadata, and related fields.
- **Replace OBP import:** A **Prism API** pull into Taylor replaces today’s ClientSpace **OBP import** for pulling current benefits. Note: today’s OBP import requires **benefit rules** and **rate groups** to exist in **both** ClientSpace and Prism; validation and migration implications are tracked in [2.4.2](./2.4.2-Pre-generation-validation-of-plans-plan-groups-and-rate-band-designation.md) and Prism adapter work ([2.2.5](./2.2.5-Adapter-mapping-layer-to-manage-Prism-schema-dependencies.md)).
- **Downstream milestones** (detail in linked tickets): workbook generation → consultation (largely unchanged) → OMQ handling where applicable → CBE / cost sheet / benefit guide → DocuSign → Prism import files → post-enrollment reconciliation/audit.

**Scale (planning):** **1,500+** clients on the **1/1** renewal wave; approximately **3–10/month** for non-1/1 renewals (varies). Use for capacity and performance expectations ([2.11.3](./2.11.3-Performance-testing-aligned-to-renewal-season-volume.md)).

### April 8, 2026

- **OMQ path:** When a client requests open-market quotes, **keep the batch in consultation**—**no new ClientSpace batch status** is required for “OMQ requested” ([2.2.2](./2.2.2-ClientSpace-API-integration-and-workflow-status-updates.md)). After quotes are keyed into the OMQ template, **regenerate** the workbook ([2.10.2.1](./2.10.2.1-Injection-of-OMQ-plans-into-renewal-workbook-alongside-Prism-sourced-plans.md)).
- **Timing hints:** **Client-sponsored** renewals often follow a **~90-day** lead; **Questco-sponsored 1/1** operations typically **start workbook batches in August** ahead of September client conversations. **Plan ID naming** distinguishes sponsorship—Questco to provide a key ([2.4.2](./2.4.2-Pre-generation-validation-of-plans-plan-groups-and-rate-band-designation.md)).
