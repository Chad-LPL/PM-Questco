# 2.3.6 - Admin console providing renewal status visibility and light reporting

## Traceability
- **Asana task ID:** `1213834655988131`
- **Section:** Backlog

## Product intent

The **Taylor admin console** is the **technical / operational plane** for the integration: pipeline health, failures, retries, processing spikes, infra-oriented visibility, and **light reporting** for engineering/ops. Day-to-day renewal **work queues**, **cases**, and **granular operational dashboards** remain **ClientSpace-first** ([2.2.2](./2.2.2-ClientSpace-API-integration-and-workflow-status-updates.md)) unless product decides otherwise.

### Product owner input (April 2026)

- **One-stop reporting:** If Questco builds a **true admin console**, Anna wants **everything** that leadership needs in **one place**—not separate pulls from Taylor **and** ClientSpace that she must **merge** for C-suite **weekly or daily** updates during open enrollment.
- **Strategic ownership:** A Questco-owned console (or Taylor-owned reporting surface) preserves **reporting** if Questco ever **left ClientSpace**—not a near-term plan, but a stated executive motivation for owning the stack.
- **Decision gate (April 8, 2026):** Questco sent required data points to **Q Insights** for evaluation. **If Q Insights cannot deliver** the needed executive/operational reporting, **proceed with the Taylor admin console** for reporting (**Dimple**, **Chris**—high priority). **Workbook generation/regeneration triggers** should **not** assume ClientSpace as the only entry point; alternatives include **manual procedures** and **Microsoft 365** automation—coordinate with [2.3.4](./2.3.4-Submission-process-for-completed-workbook-ingestion.md) and architecture review.
- **Prior open decision:** Whether to ship a **full admin console**, rely on **ClientSpace +** improvements, use **Q Insights** for executive views, or a **hybrid**—now narrowed by the Q Insights gate above. Inputs: **OMQ** handling, **rate** path ([2.5.3](./2.5.3-Rate-book-management.md)), and **reporting** needs ([2.5.2](./2.5.2-Renewal-status-dashboard.md), [2.5.6](./2.5.6-Renewal-pipeline-reporting-and-workbook-generation-activity-summaries.md)).

Prior framing still applies: default posture is **lean into ClientSpace** for assignments and workflow unless a decision record moves reporting or queues into Taylor ([WeeklyStatusCall-4.1.26.md](../../meetings/WeeklyStatusCall-4.1.26.md)).

### April 10, 2026 — Prism Open Enrollment grid visibility

- Stakeholders asked Taylor (or reporting) to **surface whether clients have Open Enrollment workflow windows / grid rows** configured in Prism so **no client is left behind** without an employee enrollment window—may be **read-only visibility** from Prism ([2.8.2](./2.8.2-Generate-Open-Enrollment-workflow-to-invoke-import.md)) rather than full automation in v1.

## Product requirements (verifiable)

- Renewal / pipeline **status visibility** and **light reporting** appropriate to operators/engineering (filters, exports TBD per SOW).
