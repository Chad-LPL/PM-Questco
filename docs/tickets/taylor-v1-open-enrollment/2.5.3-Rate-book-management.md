## Product intent

Upload, tag, and apply rate data so workbook generation and Prism stay aligned. This supports Taylor V1 open enrollment orchestration. Capabilities may live in an admin surface, **Prism-first imports**, or Taylor-orchestrated writes—see **open decision** in [2.4.1](./2.4.1-Automated-ingestion-of-client-plan-census-enrollment-data-from-prism-and-rate-data-from-admin-console-upload.md) until Questco completes the LaWanda assessment below.

### As-is process (April 2026 — Anna / Dimple)

- Carriers send **rate grids**. Upper management may **reassign** plans or apply **subsidies** before rates are “fully released.”
- **Dimple** loads plan **shells** and runs **three basic ClientSpace imports**: plan shell, plan setup information, and **rates**. Rates are entered into **ClientSpace first** (not Prism first).
- In **Prism**, at the client plan setup level, **current rate band** is populated; **future rate band** sits **empty** until the ClientSpace ↔ Prism import/export at turnover fills it from what the client is getting. At **1/1**, Prism rolls **future → current** and clears future.
- **One rate band per client** applies across all plans for that carrier book (e.g. Aetna: many plan options, **one** band such as “100” for that client).

### Target direction (discovery)

- **Avoid redundant ClientSpace rate data** where possible; **Prism is end state** for finalized renewal data.
- **Preferred path under discussion:** Import or assign the rate grid into Prism’s **future rate band** per client (possibly via Prism-native import) so Questco can **skip** much of the ClientSpace rate/shell churn. **Taylor** might orchestrate writes, or Prism import alone might suffice.
- **Open spike (Questco):** Anna/Dimple to work with **LaWanda** on what it takes to import the rate grid into the **future rate band** slot in Prism. If the lift is too high, bring the process back to LPL to scope Taylor-assisted import.

### April 8, 2026 — Prism rates vs rebanding file

- **Rates:** Team expects **future rates** to live in Prism after **Kelly/Kim** rebanding and **LaWanda**’s import (import file/columns confirmed with LaWanda). Leadership requested **published OE dates** for when all rates must be in Prism.
- **Band assignment:** Distinct from “rate book” PDFs—Taylor needs the **rebanding file** to know **which band** applies per client. That assignment is **not** stored as a clear Prism field today; stakeholders prefer **SharePoint** as the source of truth for the rebanding file so late changes do not bottleneck on Prism imports. File contains **every** client; band can change mid-season → **rerun** affected workbook.
- **Scope clarity:** This ticket still covers **operator validation** that the correct rate state is applied during generation; **SharePoint layout** for the rebanding file is [2.2.3](./2.2.3-SharePoint-file-storage-configuration.md); ingestion wiring is [2.4.1](./2.4.1-Automated-ingestion-of-client-plan-census-enrollment-data-from-prism-and-rate-data-from-admin-console-upload.md).
